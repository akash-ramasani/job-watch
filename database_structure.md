# JobWatch Firestore Database Structure

JobWatch uses a hierarchical NoSQL structure in Firebase Firestore. Below is the schema representing the current state of the database, including all collections, subcollections, and their expected fields.

---

## 1. Top-Level Collection: `users`
This is the primary collection storing all registered users.

**Path:** `/users/{uid}`
**Document Structure:**
```typescript
{
  email: string;
  firstName: string;
  lastName: string;
  fullName: string;
  createdAt: timestamp;
  lastFetchAt: timestamp | null;
  aiAccess: boolean; // Toggled by Admin to enable/disable AI features
  accountStatus: string; // "active", "trial", "pending", "deactivated"
  accountType: string; // "paid", "trial"
  trialDays: number;
  trialEndsAt: timestamp | null;
  
  // Single-session enforcement tracking
  activeSession: {
    token: string;
    deviceInfo: {
      browser: string;
      device: string;
      ip: string;
      userAgent: string;
    };
    lastSeenAt: timestamp;
    createdAt: timestamp;
  };
}
```

### Subcollection: `jobs`
Stores all the scraped/imported jobs. Most of these are stored under the Admin's UID (`/users/{ADMIN_UID}/jobs`) and served to all users.
**Path:** `/users/{uid}/jobs/{jobId}`
**Document Structure:**
```typescript
{
  title: string;
  companyName: string;
  companyKey: string;
  locationName: string;
  locationTokens: string[] | { name: string, city: string }[];
  stateCodes: string[];
  isRemote: boolean;
  jobUrl: string;
  applyUrl: string;
  source: string; // e.g., "ashby", "greenhouse"
  externalId: string;
  workplaceType: string | null;
  
  // AI Scoring Data
  relevanceScore: number | null; // e.g., 85
  scoreReason: string | null; // AI-generated rationale
  
  // Timestamps
  sourceUpdatedTs: timestamp;
  firstSeenAt: timestamp;
  fetchedAt: timestamp;
}
```

### Subcollection: `companies`
A normalized list of companies to power the filtering UI.
**Path:** `/users/{uid}/companies/{companyKey}`
**Document Structure:**
```typescript
{
  companyName: string;
  companyKey: string;
  createdAt: timestamp;
  lastJobFetchedAt: timestamp;
}
```

### Subcollection: `settings`
User-specific preferences and configuration.
**Path:** `/users/{uid}/settings/preferences`
**Document Structure:**
```typescript
{
  aiScoringEnabled: boolean;
}
```

---

## 2. Top-Level Collection: `invites`
Stores the invite codes generated by the admin to onboard new users.

**Path:** `/invites/{inviteCode}` (e.g., `/invites/ABC12345`)
**Document Structure:**
```typescript
{
  used: boolean;
  usedBy?: string; // User ID who consumed the code
  usedAt?: timestamp; // When it was consumed
  
  fullName: string; // Intended user's name
  email: string; // Intended user's exact email address
  
  accountType: string; // "paid" or "trial"
  trialDays: number; // e.g., 7 or 30
  
  expiresAt: timestamp; // Invite validity period (e.g., 12 hours from creation)
  createdAt: timestamp;
  createdBy: string; // Admin UID who created it
}
```

---

## Notes on Architecture
1. **Admin Singleton Pattern:** To save on database costs and deduplicate scraping, all jobs and companies are stored under the Admin's specific User Document (`/users/ADMIN_UID/jobs`), rather than being copied into every individual user's document. The React app dynamically reads from the Admin's subcollections to populate the feed.
2. **Session Guard:** The `activeSession` map on the User document is actively watched by the React client. If a new login overwrites the token, the older client sees the mismatch and automatically ejects the user.
3. **AI Gating:** Access to the `generateCoverLetter` Cloud Function and the visibility of `relevanceScore` is gated by the `aiAccess` boolean on the User document.
